# Wingman Memory Skills Redesign Plan

## 目标

这份文档用于后续改造四个 Wingman memory skills：

- `wingman:memory-setup`
- `wingman:memory-load`
- `wingman:memory-sync`
- `wingman:memory-clean`

核心目标不是把 memory 做成更大的日志库，而是让它更像一个可检索、可晋升、可清理的知识系统：

- `context.md` 只保存热上下文和近期任务状态，避免长期堆积。
- `domains/` 保存当前仍然有效、未来必须遵守的长期真理。
- `history/` 保存事件正文和投影索引，用于解释规则来源、演化和排查背景。
- agent 做某个功能时，能自动从相关 domain/topic 找到历史，而不是只能按“上个月做了什么”去查。

## 当前问题

当前 skill 组合的主要问题：

1. `memory-sync` 对 `DOMAIN_TRUTH` 和 `HISTORY_EVENT` 判定过保守，导致大量长期知识堆进 `context.md`。
2. `memory-load` 默认只在用户问“之前 / 上次 / history / 为什么以前”时读 history，不能在功能开发时主动加载相关历史。
3. `history/` 当前只有 domain/month projection 设计，缺少面向功能或问题簇的 topic projection。
4. `memory-clean` 能压缩 context，但没有明确“把 context 中长期知识晋升到 domain/history 后再压缩”的流程。
5. `memory-setup` 初始化模板没有把 topic projection、folder domain、promotion lifecycle 讲清楚。

## 推荐模型

### Memory 分层

```text
.wingman/memory/
  brief.md
  context.md

  domains/
    checkout.md
    ych/
      index.md
      product-detail.md
      checkout.md
      orders.md
      im.md

  history/
    index.md
    events/
      YYYY/
        MM/
          YYYY-MM-DD-<event-slug>.md
    domains/
      checkout.md
      ych.md
    topics/
      ych-product-detail.md
      ych-checkout.md
      ych-orders.md
      payment-methods.md
    months/
      YYYY-MM.md
```

### 文件职责

`brief.md`

- Memory Settings。
- 全局 ADR。
- Domain Registry。
- 不承载具体业务细节。

`context.md`

- 热上下文。
- 当前任务、pending tasks、最近少量高信号日志。
- 不作为长期知识库。

`domains/`

- 当前真理。
- API contract、字段语义、状态流、权限、金额、路由、业务规则、可复发调试结论。
- 未来 agent 必须遵守。

`history/events/`

- 事件正文，唯一完整历史记录。
- 说明“当时发生了什么、为什么这么改、涉及哪些文件、结果是什么”。
- 不作为当前规则。

`history/domains/`

- 按 domain 聚合事件链接。
- 做某个 domain 时可自动读这个 projection。

`history/topics/`

- 按功能、问题簇、工作流聚合事件链接。
- 这是支持“做某个功能时自动查相关历史”的关键入口。

`history/months/`

- 按月份聚合事件链接。
- 主要服务日期查询，例如“上个月做了什么”。
- 不作为默认功能开发入口。

## Domain 生成规则

### 什么时候创建 domain 文件

创建 `domains/<domain>.md` 或 `domains/<domain>/index.md` 的条件：

- 出现稳定业务 / 技术领域，未来多个任务会复用。
- 有当前仍然有效的规则，而不是一次性日志。
- 规则有证据来源：用户说明、接口文档、schema、测试、实现契约、已接受 spec 或现有 memory。
- 不创建会导致相关知识继续堆在 `context.md`，或 future agent 需要反复从日志中推断。

### 什么时候使用 folder domain

如果一个 domain 跨多个子流程或文件会持续增长，用 folder domain：

```text
domains/ych/index.md
domains/ych/product-detail.md
domains/ych/checkout.md
domains/ych/orders.md
```

folder domain 的 `index.md` 保存：

- `When To Read This Domain`
- 子文件路由表
- 少量跨子域通用 current truths

子文件保存具体规则。

### 推荐新增的 Domain Registry 字段

当前字段：

```text
Domain | Read When | Current File | History | Aliases | Related Domains | Status
```

建议改为：

```text
Domain | Read When | Current File | History Domain Index | History Topics | Aliases | Related Domains | Status
```

示例：

```md
| YCH | Working on YCH product detail, YCH checkout, YCH orders, YCH IM, YCH merch discovery | `.wingman/memory/domains/ych/index.md` | `.wingman/memory/history/domains/ych.md` | `ych-product-detail, ych-checkout, ych-orders, ych-im` | ych merch, commission, start request | Checkout, Product, Orders, IM | current |
```

这样 `memory-load` 可以从 registry 直接知道：

- 当前真理读哪里。
- domain 历史 projection 读哪里。
- 功能 topic projection 读哪些。

## History 机制改造

### 保留事件正文按日期存放

建议保留：

```text
history/events/YYYY/MM/YYYY-MM-DD-<event-slug>.md
```

原因：

- 事件正文路径稳定，不随 domain/topic 重组而移动。
- months projection 可以自然链接事件。
- domain/topic projection 都只是索引，可以安全重建。

### 新增 topic projection

新增：

```text
history/topics/<topic>.md
```

topic 是面向功能和问题簇的索引，例如：

- `ych-product-detail`
- `ych-checkout`
- `ych-orders`
- `ych-im`
- `payment-methods`
- `checkout-address-flow`

topic projection 模板：

```md
# <Topic> History Index

## Read When

- <task signals>

## Events

- `../events/YYYY/MM/YYYY-MM-DD-<event-slug>.md`: <one-line summary>; primary `<domain>`; promoted `<truth link or None>`
```

### history/index.md 只做入口

`history/index.md` 不应该变成完整事件列表。建议只放：

- domain projection 列表
- topic projection 列表
- month projection 列表
- 最近少量 high-signal events

### 事件正文模板调整

建议事件正文增加 `Topics` 字段：

```md
# Short Event Title

- **Date**: YYYY-MM-DD
- **Primary Domain**: `<domain or none>`
- **Related Domains**: `<domain list or None>`
- **Topics**: `<topic list or None>`
- **Type**: `feature | bugfix | refactor | debugging | contract | decision | docs | operations`
- **Files**:
  - `path/to/file`: what changed
- **Outcome**: what happened
- **Promoted Truths**:
  - `domains/<domain>.md#current-truths` or `None`
- **Notes**: historical details only
```

## memory-setup 改动方案

### 需要改的核心

`memory-setup` 应继续只创建 core files，不默认创建 `domains/` 和 `history/`，但模板要描述新版 on-demand shape。

### 修改点

1. `Brief Template` 的 Domain Registry 增加：

```text
History Domain Index
History Topics
```

2. `Memory Layout` 改成：

```md
- `domains/`: current durable domain truth, created on demand.
- `history/events/`: event bodies, created on demand.
- `history/domains/`: domain projections for feature-time lookup.
- `history/topics/`: topic projections for feature/problem lookup.
- `history/months/`: date projections for date-based lookup.
```

3. `On-Demand Domain Shape` 增加 folder domain 示例：

```text
domains/<domain>/index.md
domains/<domain>/<topic>.md
```

4. `On-Demand History Shape` 增加 `topics/`：

```text
history/
  index.md
  domains/<domain>.md
  topics/<topic>.md
  months/YYYY-MM.md
  events/YYYY/MM/YYYY-MM-DD-<event-slug>.md
```

5. 增加一句规则：

```md
History projections are lookup indexes. Event bodies remain under `events/YYYY/MM/`; do not move event bodies into domain or topic folders.
```

### 不建议改的点

- 不要让 setup 默认创建 `domains/` 或 `history/`，否则空文件会增加噪音。
- 不要让 setup 扫描代码自动推断 domain，初始化阶段证据不足。

## memory-load 改动方案

### 需要改的核心

`memory-load` 要从“只有用户问历史才读 history”，改为：

- 当前真理仍优先。
- 做功能时，如果 task 命中 domain/topic，先读相关 history projection，再选择少量事件正文。
- 月份索引只服务日期查询，不作为默认入口。

### 新的读取顺序

建议 Load Protocol 改为：

1. 检查 repository memory state。
2. 判断是否需要 memory。
3. 读 `brief.md`。
4. 读 `context.md`。
5. 用 Domain Registry 匹配 current domains。
6. 读匹配 domain 的 current files。
7. 从 registry 的 `History Domain Index` 和 `History Topics` 选择 projection。
8. 当以下任一条件成立时读 history projection：
   - 用户问历史、之前、为什么。
   - 当前任务命中 domain，且该 domain 有 history projection。
   - 当前任务命中 topic，且该 topic 有 history projection。
   - 当前 domain truth 的 `History` 字段指向具体 event。
   - context log 提到一个相关 historical event。
9. 只读 projection，不直接扫描 `history/events/`。
10. 从 projection 中选择与当前任务最相关的 0-3 个 event bodies。
11. 若 event 与 current truth 冲突，以 current truth 为准，并标记 memory 可能需要 clean。

### History projection 选择规则

优先级：

1. Direct event link from current truth。
2. `history/topics/<topic>.md`。
3. `history/domains/<domain>.md`。
4. `history/months/YYYY-MM.md`，仅当用户按日期问。
5. `history/index.md`，仅当 projection 路由不清楚或用户问 broadly。

### Context pressure 处理

`memory-load` 仍保持 read-only，但要更积极提示：

- `context.md` 超过阈值，例如 300 行，提示 memory-clean。
- 如果 context 中多条日志明显属于长期规则，但 domains/history 没承接，提示“promotion cleanup”。
- 不在 load 中改文件。

### Checklist 增加项

内部 Memory Context Checklist 增加：

- Relevant history projections read。
- Relevant history event bodies read。
- Whether context logs should be promoted or cleaned。

## memory-sync 改动方案

### 需要改的核心

`memory-sync` 要从“写最小更新”升级为“路由 + 晋升 + 冷存档”：

- 小事仍可 IGNORE。
- 热进展写 CONTEXT_LOG。
- 长期规则写 DOMAIN_TRUTH / PROJECT_ADR。
- 重要事件写 HISTORY_EVENT。
- 若 context 已经有同 feature 多条日志，下一次 sync 要考虑 promotion。

### Routing 增加一个阶段

在 Routing 前增加 `Promotion Check`：

```text
Before writing a context log, check whether the new fact or existing same-feature context logs should be promoted to domain truth or history.
```

触发 promotion 的信号：

- 同一 feature/domain 已有 2 条以上 context logs。
- 出现 API contract、schema、字段语义、枚举、payload、状态映射。
- 用户纠正了业务含义或字段含义。
- 修复了会复发的误判或调试结论。
- 跨页面、跨模块、跨 API。
- 影响 checkout、payment、orders、permissions、money、state flow。
- 未来 agent 不知道会高概率踩坑。

### DOMAIN_TRUTH 阈值调宽

原规则偏保守。建议改成：

写 `DOMAIN_TRUTH` when future agents would need the fact to avoid re-reading old logs or re-debugging:

- stable API path/body/response/field mapping
- state/status mapping
- route/query state rule
- permission/payment/money rule
- recurring debugging conclusion
- product/business invariant
- cross-file behavior contract

### HISTORY_EVENT 阈值调宽

建议从 `defaults to no` 改成：

```md
HISTORY_EVENT defaults to no for small local changes, but defaults to yes when DOMAIN_TRUTH or PROJECT_ADR was written for a non-trivial feature, contract, state-flow correction, or recurring debugging conclusion.
```

也就是说：

- 写 domain truth 不一定都写 history。
- 但如果 domain truth 是由一次重要事件形成的，要写 history event。

建议 history event 写入条件：

- 有 current truth 被 promoted，并且需要解释来源。
- bug/regression/state-flow correction。
- API contract/schema/field decision。
- complex debugging conclusion。
- feature milestone that changes data flow across modules。
- user explicitly asks to remember historical context。

### context 写入策略

如果同时写 domain truth/history：

- `context.md` 只写短指针，不重复完整事件。
- 格式：

```md
### [YYYY-MM-DD] <short title>

- **目标**: <one sentence>
- **结果**: Current truth/history 已写入。
- **Pointers**:
  - `domains/...`
  - `history/events/...`
```

如果是普通短期进展，仍用现有 context log 模板。

### domain 写入策略

当 domain 文件不存在：

1. 判断 file domain 还是 folder domain。
2. 创建 domain 文件或 folder index。
3. 更新 `brief.md` Domain Registry。
4. 如果写了 history event，domain truth 的 `History` 填具体 event；否则 `None`。

当 domain 存在但太宽：

- 不继续往大文件塞。
- 创建 folder domain 或 topic subfile。
- 更新 registry。

### history 写入策略

写 history event 时更新：

- `history/events/YYYY/MM/YYYY-MM-DD-<event-slug>.md`
- `history/domains/<primary-domain>.md`
- `history/topics/<topic>.md`
- `history/months/YYYY-MM.md`
- `history/index.md`

projection 中只写一行摘要和链接，不复制正文。

### 报告输出

sync 完成后报告：

- context 是否写入。
- domain truth 是否写入/更新。
- history event 是否写入。
- 更新了哪些 projection。
- 如果没写 history，说明哪个 threshold 阻止。

## memory-clean 改动方案

### 需要改的核心

`memory-clean` 不只是压缩 context，还要支持“promotion cleanup”：

- 从 context 中识别长期知识。
- 先晋升到 domains/history。
- 再压缩或删除 context 中已被承接的日志。

### 新增 scope

建议 Scope Selection 增加：

| Scope                 | Read                                                           | Use When                                                                          |
| --------------------- | -------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| `promotion`           | `context.md`, `brief.md`, relevant domains/history projections | Context contains long-lived knowledge that should become domain truth or history. |
| `domain-split`        | `brief.md`, one broad domain file                              | A domain file mixes unrelated subdomains and should become folder domain.         |
| `history-topic-index` | relevant history files                                         | History exists but feature lookup needs topic projections.                        |

### 新增 cleanup workflow

当用户说“memory 太长 / context 太多 / 整理 memory / 压缩记忆”时：

1. 默认先检查 `context.md` 行数。
2. 如果 context 里有长期知识，进入 `promotion`。
3. 分类每条日志：
   - `KEEP_HOT`
   - `PROMOTE_DOMAIN`
   - `PROMOTE_HISTORY`
   - `PROMOTE_BOTH`
   - `COMPACT_TO_POINTER`
   - `DELETE_CANDIDATE`
4. 对 `PROMOTE_*`：
   - 写 domain/history。
   - 将 context 日志压缩为 pointer。
5. 对 `DELETE_CANDIDATE`：
   - 仍然需要单独确认。

### 删除规则保持严格

保持原规则：

- 不经确认不删除。
- 不删除 current truth。
- 不删除 history event body，除非用户明确点名。

但可以自动做：

- lossless compaction。
- supersede。
- context log -> pointer。

### memory-clean 与 memory-sync 的关系

- `memory-sync` 是增量路由。
- `memory-clean` 是事后整理。
- 如果 `memory-sync` 做得好，`memory-clean` 只需要偶尔运行。
- 如果旧仓库已有很多 context，`memory-clean promotion` 是迁移工具。

## 四个 skill 之间的协作

### setup

只建立能力和模板，不生成空 domain/history。

### load

读取当前真理，然后按 domain/topic 自动选择历史 projection。

### sync

增量记录，并把长期知识及时晋升到 domain/history。

### clean

迁移旧 context，压缩默认读取成本，修复过宽 domain 和缺失 projection。

## 推荐迁移步骤

后续实际改 skill 时建议按这个顺序：

1. 改 `memory-setup` 模板，加入新版 registry/history shape。
2. 改 `memory-sync`，加入 promotion check、topic projection、放宽 domain/history 阈值。
3. 改 `memory-load`，让开发任务按 domain/topic 自动读 history projection。
4. 改 `memory-clean`，加入 promotion cleanup、domain split、history topic index。
5. 用当前 `acggoods` memory 做一次试迁移：
   - 新建 `domains/ych/`。
   - 从 context 提炼 YCH product/checkout/orders/im current truths。
   - 为关键事件补 history events。
   - 建 `history/topics/ych-*`。
   - 将 context 旧日志压缩成 pointers。

## 需要验证的场景

后续改完 skill 后，至少用这些任务做 forward test：

1. 用户要求改 YCH 订单进度条。
   - `memory-load` 应读 `domains/ych/orders.md` 和 `history/topics/ych-orders.md`。
2. 用户要求改 YCH checkout create order payload。
   - `memory-load` 应读 `domains/ych/checkout.md`、`domains/checkout.md` 和相关 history topic。
3. 用户只做小视觉图片替换。
   - `memory-sync` 应 IGNORE 或只写短 context，不写 domain/history。
4. 用户修复支付双选 bug。
   - `memory-sync` 应写 domain truth 和 history event。
5. 用户要求清理 memory。
   - `memory-clean` 应先 promotion，再 compact context，不直接删除日志。

## 当前不确定点

这些点建议你在正式改 skill 前确认：

1. 是否接受新增 `history/topics/` 目录。
   - 推荐接受。
   - 如果不接受，也可以把 topic projection 放到 `history/domains/<domain>/<topic>.md`，但 lookup 规则会稍微绕。

2. 是否接受 folder domain 作为常规结构。
   - 推荐接受，尤其适合 `YCH` 这种跨 product/checkout/orders/im 的长期功能。

3. Domain Registry 是否可以改 schema。
   - 推荐加 `History Domain Index` 和 `History Topics`。
   - 如果需要兼容旧 registry，`memory-load` 要同时支持旧 `History` 字段和新字段。

4. `HISTORY_EVENT` 默认是否要从 “defaults to no” 改成更积极。
   - 推荐改为：“小改默认 no；写入非平凡 DOMAIN_TRUTH/PROJECT_ADR 时默认 yes，除非缺少事件价值。”

5. `context.md` 目标长度阈值是多少。
   - 建议默认软阈值 200 行，硬阈值 400 行。
   - 也可以在 `brief.md` Memory Settings 里配置。

6. 是否允许 `memory-clean` 在无需逐条确认的情况下做 context log -> pointer 的 lossless compaction。
   - 推荐允许。
   - 删除仍必须单独确认。

7. 是否要提供迁移脚本。
   - 推荐先不写脚本，先把 skill 规则跑顺。
   - 等结构稳定后，再考虑脚本生成 memory stats 或 projection consistency check。
