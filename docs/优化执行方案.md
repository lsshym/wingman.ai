# Wingman memory 优化执行方案

本文记录 Wingman memory 重设计的执行方案。目标是把当前 memory 从“几个 Markdown 文件 + 规则约定”升级成一个职责清晰、可渐进实现、可长期维护的项目记忆系统。

相关文档：

- `docs/Wingman memory 设计问题分析.md`
- `docs/Wingman memory 参考模型与解决方案.md`
- `docs/待优化项.md`

## 执行约定

- 后续手动修改只处理主源码、文档、测试和 eval。
- 不再手动同步 `plugins/wingman/` payload 目录；该目录由维护者在合适时机统一同步。
- 第一轮误同步到 `plugins/wingman/` 的改动已按用户要求做文件层面还原；后续步骤不再继续编辑该目录，避免主源码和 payload 在分阶段改造中反复产生半同步状态。
- 每完成一个步骤，都要在本文对应步骤下追加“执行记录”，说明改了什么、行为变化、未处理事项和是否运行测试。
- 每块改动开始前，必须读取该块“参考什么方案”里列出的相关原文或官方文档；读不到原文时必须在执行记录里标明“未核验原文”，并且不能把未核验摘要当作设计依据。
- 每块执行记录必须补充“参考原文核验记录”，至少写清楚读了哪些链接、从原文采用了哪些原则、哪些参考只保留为待核验。
- 测试用例文件暂时不由 agent 修改；全部功能和文档改完后，由用户统一修改测试文件、eval cases、eval fixtures 和测试断言。每个待优化项仍然要在执行记录里详细写清楚应如何改测试、改哪里、为什么这么改、行为有什么变化，方便后续统一同步测试。
- 不修改 `plugins/wingman/` 下内容；用户会手动同步。之前误改的 payload 已按文件层面还原；后续不再动。
- 暂时不跑测试；等全部改完后统一跑。

## 0. 总体方向

本次优化会参考多个成熟模型，但不会把它们全部照搬进 Wingman。采用原则如下：

| 参考模型 | 借鉴内容 | Wingman 中的用途 |
| --- | --- | --- |
| Repo-scoped opt-in tools | 工具能力安装与仓库配置启用分离 | 定义 `memory installed / disabled / enabled / partial` 状态 |
| LangGraph memory | short-term / long-term、semantic / episodic / procedural 分类 | 定义 `context`、`domains`、`history`、`brief` 的职责 |
| Letta / MemGPT | core memory vs recall / archival memory，按需检索 | 定义哪些 memory 默认读，哪些只在用户问历史时读 |
| Zep / Graphiti | temporal knowledge graph、事实状态、历史上下文 | 给 domain truth 增加 `Status`、`Since`、`Supersedes`、`Related Domains` |
| ADR | 架构决策状态和替代关系 | 规范全局架构决策和 memory 结构决策 |
| Cline Memory Bank | Markdown 文件落地、人类可读 | 保留项目内 Markdown memory，但避免全部默认读取 |
| Generative Agents | 从事件流 reflection 出高层记忆 | 让 `memory-sync` 从 active log 中提炼 domain truth |
| Reflexion | 从失败反馈中形成可复用经验 | 把测试失败、调试结论升格为有证据的 domain truth |
| CoALA | memory 与 action 分离 | 把 setup/load/sync/history maintenance 的动作边界写清楚 |

最终目标结构倾向于：

```text
.wingman/memory/
  brief.md
  context.md
  domains/
    <domain>.md
    <domain>/index.md
  history/
    index.md
    events/
      YYYY-MM-DD-<event-slug>.md
```

这里的核心边界是：

- `brief.md`：全局规则、ADR、memory settings、Domain Registry。
- `context.md`：当前任务、短期上下文、最近高信号日志。
- `domains/`：当前有效的领域规则、契约、状态流、坑点。
- `history/`：历史事件和过去工作记录，默认不作为当前规则读取。

## 1. 先解决 memory enabled 状态

### 修改什么问题

当前插件安装后，agent 拥有 `memory-load` skill，但仓库不一定有 `.wingman/memory/`。这导致普通非平凡任务可能触发 memory-load，然后发现没有 memory root 再安静跳过。

这会造成体验混乱：

- 用户以为项目已经启用了 memory。
- agent 展示了 memory 流程，但没有读到任何 memory 文件。
- `memory-load` 和 `memory-sync` 对缺少 root 的处理不一致。

### 参考什么方案

本步骤主要参考 repo-scoped opt-in tools，而不是直接参考 agent memory taxonomy。

更准确的类比是：

- pre-commit：工具可以安装在用户环境里，但仓库需要 `.pre-commit-config.yaml` 才算配置了 pre-commit hooks。缺少配置时，普通工作不应表现得像仓库已经启用了 hooks。
- GitHub Actions：平台能力存在，但只有仓库中存在 `.github/workflows/*.yml`，事件才有实际 workflow 可执行。
- Git LFS：用户级 `git lfs install` 只安装能力；仓库级 `.gitattributes` 才声明哪些文件真正由 LFS 管理。

这些模型都区分：

```text
Capability installed  !=  Repository enabled
```

这比用 LangGraph / Letta 直接证明 enabled gate 更合适。Wingman plugin installed 只是说明 agent 拥有 `memory-load` / `memory-sync` / `memory-setup` 能力；当前仓库是否启用 memory，应该由仓库内的 `.wingman/memory/` 结构决定。

Letta 和 LangGraph 只作为辅助参考：

- Letta memory blocks 是显式创建/附着的上下文资源，支持“memory 不是安装框架后天然存在的隐式数据”这个类比。
- Letta archival memory 是按需查询的长期存储，支持后续把 `history` 设计成默认不读取的冷历史。
- LangGraph 的 short-term / long-term、semantic / episodic / procedural taxonomy 适合支持后续 memory 类型拆分，但不作为本步骤 enabled/disabled gate 的直接依据。

参考链接：

- pre-commit docs: https://pre-commit.com/
- GitHub Actions workflows: https://docs.github.com/en/actions/concepts/workflows-and-actions/workflows
- Git LFS docs: https://git-lfs.com/
- Letta memory blocks: https://docs.letta.com/guides/core-concepts/memory/memory-blocks/
- Letta archival memory: https://docs.letta.com/guides/core-concepts/memory/archival-memory/
- LangGraph / LangChain memory overview: https://docs.langchain.com/oss/javascript/concepts/memory

### 准备怎么改

建立四个状态，避免把“插件可用”和“仓库已启用”混在一起：

```text
memory capability installed
  = Wingman plugin 提供 memory-load / memory-sync / memory-setup

repository memory disabled
  = 当前仓库没有 .wingman/memory/

repository memory enabled
  = .wingman/memory/ 存在，且核心入口文件存在

repository memory partial / broken
  = .wingman/memory/ 存在，但核心入口文件缺失或结构不完整
```

调整规则：

1. `using-wingman` 先判断 repository memory state，再决定普通任务是否触发 `memory-load` / `memory-sync`。
2. `memory-load` 的普通触发条件必须包含“repository memory enabled”。
3. `memory-sync` 的普通完成后触发条件也必须包含“repository memory enabled”。
4. 仓库没有 `.wingman/memory/` 时，普通任务不声明或执行 `memory-load` / `memory-sync`。
5. 用户明确问 memory、历史、保持一致、要求 setup、或明确要求 sync/update memory 时，才说明 memory 未启用。
6. `memory-setup` 是唯一从 disabled 进入 enabled 的入口。
7. `memory-load` / `memory-sync` 都不能把缺少 root 当作自动创建 memory 的理由。
8. `memory-load` / `memory-sync` 内部保留 defensive fallback：如果被直接调用但发现 root 缺失或结构不完整，按 disabled / partial 状态处理，不创建文件，不把空目录当成可用 memory。
9. `repository memory partial / broken` 只在用户问 memory、明确要求 sync/update memory、或任务必须依赖历史一致性时提示；普通任务不因为半残结构而进入 memory 流程。

## 2. 简化 memory-setup 初始文件

### 修改什么问题

当前 setup 会创建：

```text
projectBrief.md
activeContext.md
domains/README.md
archive/README.md
```

其中两个 README 是目录规则说明，不是真正的 memory 内容，而且规则已经存在于 skill 中。

问题：

- 初始文件偏多。
- README 和 skill 规则重复。
- 未来容易出现 README 规则和 skill 规则漂移。

### 参考什么方案

本步骤主要参考 minimal initializer / lazy materialization / single source of truth，而不是直接照搬某个 memory bank 的文件数量。

更准确的类比是：

- `git init`：初始化只创建让仓库成立的最小结构，不替用户预创建业务目录。
- `npm init`：初始化只创建 package metadata 的核心入口，不预生成 `src/`、`docs/`、`tests/` 等未来可能需要的目录。
- Git LFS：用户级能力安装后，只有明确 track 某类文件时才写入仓库级 `.gitattributes` 规则。

这些模型支持 Wingman 把 `memory-setup` 设计成“启用 memory 的最小入口”，而不是一次性生成所有未来可能用到的目录和说明文件。

Cline Memory Bank 只作为辅助参考：

- memory 文件可以使用项目内 Markdown，方便人类直接阅读。
- 每个文件必须有明确职责。
- 当前上下文文件应该保持短小，不应该变成完整 changelog。
- 但 Cline 自身的多 core files 不作为 Wingman setup seed 数量的依据。

参考链接：

- Git init: https://git-scm.com/docs/git-init
- npm init: https://docs.npmjs.com/cli/v8/commands/npm-init
- Git LFS track: https://github.com/git-lfs/git-lfs/blob/main/docs/man/git-lfs-track.adoc
- Cline Memory Bank: https://docs.cline.bot/prompting/cline-memory-bank
- Cline Memory Bank alternate entry: https://docs.cline.bot/features/memory-bank

### 准备怎么改

setup 默认只创建 memory enabled 的最小核心入口：

```text
.wingman/memory/
  brief.md
  context.md
```

不再默认创建：

```text
domains/README.md
history/README.md
```

目录按需创建：

```text
domains/      # 第一次写入 domain truth 时创建
history/      # 第一次写入历史事件时创建
```

如果为了让目录自解释，可以在 `brief.md` 中保留“Memory Layout”段，但它只能是目录地图，不能复制 `memory-load` / `memory-sync` 的执行规则，避免再次产生规则双写和漂移。

核心原则：

- `brief.md` + `context.md` 是 memory enabled 的最小判断入口。
- `domains/` 是第一次写入当前有效 domain truth 时才创建的存储区。
- `history/` 是第一次写入历史事件时才创建的存储区。
- 规则来源保持单一：执行规则在 skill 中，memory 文件保存项目自己的记忆。

## 4. 重建 memory 类型边界

### 修改什么问题

当前 memory 的类型边界还不够硬。即使第 2、3 条已经把文件名改成 `brief.md`、`context.md`、`domains/`、`history/`，仍然需要明确每类文件到底保存什么、默认读什么、按需查什么。

核心问题不是“有没有分类名”，而是：

- 当前状态和过去事件容易混在一起。
- 当前有效规则和历史上曾经采用过的方案容易混在一起。
- 短期任务上下文和长期领域规则容易混在一起。
- `history/` 如果被默认当作当前规则读取，旧结论会污染当前实现。

### 参考什么方案

本步骤不再只参考 LangGraph memory taxonomy。LangGraph 的分类适合命名，但不足以单独解决 Wingman 的文件边界和读取优先级问题。

采用组合模型：

| 参考模型 | 借鉴内容 | Wingman 中的用途 |
| --- | --- | --- |
| LangGraph memory taxonomy | short-term / long-term、semantic / episodic / procedural | 给 `context`、`domains`、`history`、`brief` 命名和分工 |
| Event Sourcing / audit log | current state 和 event log 分离 | 区分“当前有效状态”与“过去发生过的事件” |
| ADR | decision status、context、decision、consequence、supersede | 区分“当前有效决策”和“历史上曾经做过的选择” |
| Letta / MemGPT | context 中默认可见的 core memory 与按需查询的 archival memory 分层 | 决定哪些 memory 默认读取，哪些只在用户问历史或需要溯源时读取 |

具体采用：

- LangGraph 只负责术语层的分类：
  - `context.md` 对应 short-term memory。
  - `domains/` 对应 semantic memory 和领域 procedural memory。
  - `history/` 对应 episodic memory。
  - `brief.md` 对应全局 procedural memory、ADR 索引和 memory routing。
- Event Sourcing / audit log 负责硬边界：
  - `domains/` 和 `brief.md` 保存当前有效状态。
  - `history/` 保存过去事件、变更原因和溯源材料。
  - 历史事件可以解释“为什么变成这样”，但不能自动覆盖当前状态。
- ADR 负责决策生命周期：
  - 当前有效决策必须有明确状态。
  - 被替换的方案必须标记 superseded / deprecated，而不是继续和当前规则平级。
  - 需要把“曾经考虑过的替代方案”与“现在必须遵守的决策”分开。
- Letta / MemGPT 负责读取策略：
  - 小而关键、必须一直可见的内容放在默认读取路径。
  - 大量历史、旧日志、过去事件放在按需查询路径。
  - `history/` 默认不读全文，只在用户问历史、排查回归、需要解释来源时读取索引和相关事件。

字段含义：

- `Domain`：规范 domain 名。
- `Read When`：任务里出现哪些线索时读取。
- `Current File`：当前 truth 文件。
- `History`：history index 查询入口，不是当前规则；不能因为这里有链接就默认读取所有 history event。
- `Aliases`：旧名称、同义名称。
- `Related Domains`：跨域任务可能要考虑的 domain；只有和当前任务相关时才读取，不能因为 related 就全量读取。
- `Status`：domain 是否仍作为读取路由使用，取值为 `current | deprecated | superseded`。

状态语义必须拆开：

```text
Domain Registry Status
  = domain 路由是否仍然有效
  = current | deprecated | superseded

Domain Truth Status
  = 某条领域规则是否仍然能约束未来实现
  = current | deprecated | superseded

ADR Status
  = 架构决策生命周期
  = proposed | accepted | deprecated | superseded
```

其中：

- `Domain Registry Status: current` 表示这个 domain 仍用于路由。
- `Domain Truth Status: current` 表示这条领域规则能作为当前实现依据。
- `ADR Status: accepted` 表示这条架构决策已经生效；不要把 ADR 的 `accepted` 和 domain truth 的 `current` 混成同一套状态。

domain truth 模板统一为：

```markdown
<domain.md section: Current Truths>

- `<rule>` [WHY]: `<reason>`
  - **Evidence**: `<user statement | docs/schema/tests/spec | implementation>`
  - **Applies When**: `<future task condition>`
  - **Status**: `current | deprecated | superseded`
  - **Since**: `YYYY-MM-DD`
  - **Supersedes**: `<old rule or None>`
  - **Related Domains**: `<domain list or None>`
  - **History**: `<history/events/YYYY/MM/YYYY-MM-DD-<event-slug>.md or None>`
```

规则：

- `Status: current` 的 truth 才能作为未来实现依据。
- 被替代规则不能无痕删除，应该标记 `superseded` 或 `deprecated`，并尽量链接到替代规则或 history event。
- 如果同一 domain 有冲突的 current truth，`memory-sync` 必须停下来解决冲突，不能继续追加新冲突。
- Domain Registry 中的 `Related Domains` 只负责提示跨域读取，不代表把跨域 truth 复制到多个 domain。
- `History` 是可选回链字段，用于说明这条 truth 来自哪个历史事件；没有对应事件时写 `None`，不能为了填字段而编造 history event。
- `History` 回链不能替代当前 truth 正文；未来实现必须仍以 `Status: current` 的规则正文为准。

`memory-load` 使用顺序：

```text
1. 读 brief.md
2. 读 context.md
3. 用 Domain Registry 匹配任务
4. 读 relevant domains
5. 只有用户问历史才读 history index 和命中事件
```

## 6. 合并 history event store 和 memory-sync distillation

### 修改什么问题

按月 archive：

```text
archive/2026-05.md
```

不适合回答：

```text
upload 模块之前做过什么？
```

按业务 archive：

```text
archive/upload.md
```

又会继承 domain 边界不稳定问题。

同时，原方案把“history 改成 index + event files”和“重写 memory-sync 流程”拆开，但这两件事实际强绑定：

- `memory-sync` 是写 history event 的入口。
- history event 的 `Promoted Truths` 依赖 `memory-sync` 先判断是否要提升 current truth。
- context 维护、domain truth 提炼、history event 写入需要同一个阶段顺序，否则容易重复写或漏写。

### 参考什么方案

本步骤主要采用 Event Sourcing + CQRS projections / read models，而不是继续在“按月 archive”和“按业务 archive”之间二选一。

核心模型：

```text
event body = single source of truth
domain/month indexes = projections / read models
```

采用原则：

- Event Sourcing：历史正文是事件序列，只追加，不按查询方式复制。
- CQRS read model / projection：按业务、月份等查询方式建立索引；索引是从事件正文派生出来的读取视图。
- Generative Agents：事件流可以通过 reflection 提炼出高层记忆，但提炼后的 current truth 应写入 `domains/` 或 `brief.md`。
- Zep / Graphiti：episode 可以和 domain/entity/fact 关联，但 episode 不等于当前 fact。
- Reflexion：失败反馈、测试失败、调试结论可以形成可复用经验，但必须有触发条件和证据。

参考链接：

- Event Sourcing: https://martinfowler.com/eaaDev/EventSourcing.html
- CQRS pattern: https://learn.microsoft.com/en-us/azure/architecture/patterns/cqrs
- Generative Agents paper: https://huggingface.co/papers/2304.03442
- Graphiti overview: https://help.getzep.com/graphiti/graphiti/overview
- Reflexion paper: https://huggingface.co/papers/2303.11366
- Reflexion publication page: https://collaborate.princeton.edu/en/publications/reflexion-language-agents-with-verbal-reinforcement-learning-2

### 准备怎么改

history 采用：

```text
history/
  index.md
  domains/
    <domain>.md
  months/
    YYYY-MM.md
  events/
    YYYY/
      MM/
        YYYY-MM-DD-<event-slug>.md
```

含义：

- `history/events/YYYY/MM/*.md`：事件正文，只存一份，是历史事件的 single source of truth。
- `history/domains/<domain>.md`：按 domain 查询历史的 projection / read model。
- `history/months/YYYY-MM.md`：按月份查询历史的 projection / read model。
- `history/index.md`：总入口，只列 domain/month projection 入口和少量最近事件，不无限增长。

不采用：

```text
history/events/upload/...
```

原因：一个事件经常跨多个业务 domain。事件正文如果按业务存放，会遇到“跨域事件放哪”和“复制多份导致漂移”的问题。事件正文应按时间分区，业务视角通过 `history/domains/*.md` 投影解决。

```markdown
- **Date**: YYYY-MM-DD
- **Primary Domain**: upload

- **Files**:
  - `path/to/file`: what changed
- **Outcome**: what happened
- **Promoted Truths**:

- **Notes**: historical details only
```

```markdown
# History Index

<section: Domain Indexes>
- `domains/upload.md`: upload history projection

<section: Month Indexes>
- `months/2026-05.md`: 2026-05 history projection

<section: Recent High-Signal Events>

```

`history/domains/upload.md` 模板：

```markdown
# Upload History Index

<section: Events>
- `../events/2026/05/2026-05-13-upload-worker-retry.md`: retry handling for upload worker; related `task-queue`; promoted `domains/upload.md#current-truths`
```

`history/months/2026-05.md` 模板：

```markdown

<section: Events>
- `../events/2026/05/2026-05-13-upload-worker-retry.md`: upload worker retry handling; primary `upload`
```

这样可以同时回答：

- “upload 之前做过什么？”
- “上个月做过什么？”

- `context.md` 记录短期进展。

行为变化：

- 解决旧的两难：
  - 按月 archive 不好回答“upload 之前做过什么”。
  - 按业务 archive 又会让跨域事件复制或边界不稳定。
- 新模型下，历史事件正文只存一份；按业务、按月份查询都通过 projection index 解决。

- `memory-load` 问历史时不会全局搜索所有 event，而是走：
  ```text

    -> matched history/events/YYYY/MM/*.md
  ```
- `history/index.md` 不再承担无限增长的完整事件列表，只是入口和少量最近高信号事件。

为什么这么改：

- Event Sourcing 适合定义历史事件正文：事件记录过去发生过什么，是历史事实来源。
- CQRS projections/read models 适合定义 `history/domains/*.md` 和 `history/months/*.md`：它们是为查询服务的索引，不是事实正文。
- Generative Agents 支持从事件流 reflection 出高层记忆，但提升后的 current truth 仍必须写入 `domains/` 或 `brief.md`。
- Graphiti 支持 episode 和 domain/entity/fact 关联，但 episode 不等于当前 fact。
- Reflexion 支持把有证据的失败反馈/调试结论变成可复用经验，但普通失败尝试不能升格。

### 参考什么方案

本步骤主要参考工程职责拆分模型，而不是继续引入新的 memory 内容模型。第 1-6 条已经解决“memory 应该是什么”；第 7 条只解决“这些规则应该放在哪个 skill，哪些重复说明应该删掉”。

采用组合模型：

| 参考模型 | 借鉴内容 | Wingman 中的用途 |
| --- | --- | --- |
| Single Responsibility Principle / Separation of Concerns | 一个模块只承担一个清晰职责，变化原因要尽量单一 | 每个 skill 只保留自己最接近执行动作的规则 |
| CoALA action space | 把 agent 的 memory、reasoning、external action 拆成不同动作类型 | 明确 `memory-setup`、`memory-load`、`memory-sync`、`using-wingman` 的动作边界 |
| CQRS | read model 和 write model 分离，各自按用途优化 | `memory-load` 保持读侧规则，`memory-sync` 保持写侧和 projection 维护规则 |
| Single Source of Truth / DRY | 一个行为规则只在一个权威位置完整展开 | 避免 README、setup 模板、using-wingman 重复复制 skill 内部流程 |
| Letta context hierarchy / archival memory | 小而关键内容默认可见，大量历史按需检索 | 支撑“入口文档保持短，历史细节不塞进默认上下文”的瘦身方向 |

具体采用：

- 用 Single Responsibility / Separation of Concerns 判断“这段规则是否属于这个 skill”。
- 用 CoALA 判断“这是 setup 动作、read 动作、write 动作，还是入口调度动作”。
- 用 CQRS 判断“这是读取策略，还是写入/投影维护策略”，避免 `memory-load` 和 `memory-sync` 互相复制。
- 用 Single Source of Truth / DRY 判断“这段话是否已经在更权威的位置完整写过”，重复内容只保留短引用。
- 用 Letta context hierarchy 作为辅助依据，控制 README、`using-wingman`、setup 模板的上下文体积。

行为变化：

- memory eval 从“Node 脚本硬编码断言具体文案和旧路径”改成“case 文档定义测试规范，runner 生成执行产物，AI reviewer 按重点判断是否通过”。
- `memory-runner.test.mjs` 只保障基础机制：case 表能解析、prompt 不泄露重点、fixture 结构有效、dry-run 能生成 summary/comparison/prompt/review 输入数据。
- fixtures 已从旧 `projectBrief.md` / `activeContext.md` / `archive/` 结构迁移到 `brief.md` / `context.md` / `domains/` / `history/events/YYYY/MM/` 新结构。
- 测试重点会更直接覆盖前 1-7 条优化后的行为：enabled gate、最小 setup、current truth 优先、Domain Registry routing、history projection、sync 写入边界和 skill 瘦身后的职责边界。

为什么这么改：

- 当前测试/eval 确实写死旧路径和旧 routing 名称，如果直接跑或直接改脚本，会把已经废弃的行为重新固定下来。
- memory skill 的正确性更适合评审型 eval：很多判断依赖“是否按重点读写、是否污染无关文件、是否把 history 当 current truth”，不适合全部写成易碎的字符串断言。
- Node test 只保 eval 工具链不坏，具体通过/失败交给 AI reviewer 结合 case 重点、文件快照、最终输出判断。
- 这一步重点保证现在的 skill 符合预期，而不是保留旧 Memory Bank 风格文件结构。
