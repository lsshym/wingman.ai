# 架构决策记录 (ADR): Memory Bank v2.1 设计哲学

**日期**: 2026-04-08
**主题**: AI Agent 记忆库从“无限追加日志”向“三层动态升华架构”的演进
**状态**: 已应用 (v2.1 Strict Edition)

---

## 1. 背景与痛点 (Context & Pain Points)
在 V1 版本中，我们采用的是单文件、只追加 (Append-Only) 的记录模式（即把所有流水账都塞进 `activeContext.md`）。随着项目（如 Plushie 需求）的推进，暴露出了 AI 辅助开发中极其经典的致命问题：
1. **知识污染与精神分裂**：新旧规则（如 UI 样式单位的变更、众筹状态的流转机制）在同一文件中并存，AI 无法区分当前真理，极易重蹈覆辙。
2. **Token 爆炸与注意力涣散**：后台 (Admin)、前台 (Plushie)、结算 (Checkout) 的日志混杂。AI 在处理前台需求时，被迫读取几百行后台逻辑，导致响应变慢且频繁幻觉。
3. **AI 的“惰性描述”**：如果没有极强的外力约束，AI 会用“用于展示”、“基础组件”等正确的废话敷衍日志，导致历史记录毫无溯源价值。

## 2. 核心架构决策 (Architecture Decisions)

### 决策一：采用“三层箱庭式”路由架构 (3-Layer Dynamic Routing)
**方案**：放弃单体日志，拆分为 `Brain (全局)` -> `Domain (领域)` -> `Active Context (活跃草稿)`。
**[WHY] 为什么这么做？**
- **精准降噪**：采用类似操作系统的“按需寻址”。通过全局的 Domain Index，让 AI 在处理 Plushie 需求时，**只读取** Plushie 的真理表，绝对物理隔离 Admin 的逻辑。
- **保护 Token**：极大地精简了 AI 每次请求的上下文窗口，提升推理智商。

### 决策二：状态与日志分离 (Feature Truth Table vs. Sprint Log)
**方案**：在 Domain 文件中建立“业务真理表”（覆盖/重写模式），在 Active Context 中保留短期流水账（追加/定期清理模式）。
**[WHY] 为什么这么做？**
- **引入 Reducer 机制**：代码的演进应该是状态 (State) 的更新，而不是 Action (Log) 的无限堆砌。真理表只保留当前唯一生效的口径（例如：目前打样中就是不渲染进度条），杜绝 AI 看到旧日志后乱提建议。

### 决策三：强制“知识升华” (Knowledge Distillation)
**方案**：在 `SKILL.md` 中赋予 AI 判断力，将高价值的“业务规则”、“架构踩坑”提炼转移到 Domain/Brain 层，并定期 `PURGE`（清理）过期流水账。
**[WHY] 为什么这么做？**
- **提纯价值**：半年前改了哪个文件的变量名毫无意义，但“为什么当时放弃了方案 A 选择方案 B”价值连城。升华机制强迫 AI 把短暂的 Action 转化为长期的永久知识。

### 决策四：铸造“绝对铁律”防呆补丁 (Iron Laws & Gatekeepers)
**方案**：植入极其严苛的格式卡点（Checklist），内置“废话拦截器”（Generic Filler Check）。
**[WHY] 为什么这么做？**
- **对抗 AI 的惰性**：这是用实际开发中的血泪教训换来的紧箍咒。强制要求列出 `[Interaction]`、`[Data/State]`，直接掐断 AI 使用诸如“包含逻辑”、“支持功能”等毫无营养的套话，确保写下来的每一行字都能用于未来的 Debug。

## 3. 结论 (Conclusion)
V2.1 不是一个简单的备忘录，而是一个带有垃圾回收机制、懂得分库分表、且有着严厉自我审查机制的**“架构师级别 AI 记忆引擎”**。它确保了项目越庞大，AI 的记忆越清晰，永远不会被自己的历史日志绊倒。
