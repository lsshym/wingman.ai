### 🛑 关键格式规则

3.  **无废话**：不要输出对话式的填充内容。只输出表格。

### 🧠 智能路径检测 (目标嗅探)
在确定从哪里提取/查找类型时，严格遵循以下优先级：
1.  **现有导入**: 如果文件已经导入了类型（例如：来自 `@/types/user` 或 `./models`），复用该路径。
2.  **领域共存**: 如果不存在全局导入，建议使用特定领域的同级文件（例如：在 `features/User/types.ts` 中）。
3.  **兜底**: 仅当未发现上述模式时，建议使用通用的 `types.ts`。
4.  **约束**: 必须在表格中列出确定的路径。如果文件不存在，在路径后追加 `(New)`。

### 🔍 分析逻辑与映射
根据代码库评估每个类型/接口，并严格映射到 **状态 (Status)** 列：

1.  **逻辑：MATCH** (完美匹配)
    * *条件*: 本地类型拥有与共享类型完全一致的字段。
    * *输出状态*: **✅ 完美匹配**
    * *操作*: 合并并从共享路径引入。
2.  **逻辑：EXTRACT** (内联定义)
    * *条件*: 代码使用了内联类型（例如：在 `useState`, `props`, 或 `zod` schemas 中）。
    * *输出状态*: **📦 内联定义**
    * *操作*: 提取到检测到的目标路径。
3.  **逻辑：PARTIAL** (部分复用/子集)
    * *条件*: 本地类型扩展或缩小了共享类型。
    * *输出状态*: **🔗 部分复用**
    * *操作*: 使用 `Pick`, `Omit`, 或 `extends`。
4.  **逻辑：CONFLICT** (严重冲突)
    * *条件*: 名称相同但类型/含义不兼容。
    * *输出状态*: **🛑 命名冲突**
    * *操作*: 重命名本地类型（例如：增加组件名前缀）。
5.  **逻辑：SAFE** (忽略)
    * *条件*: 标准类型（`Record`, `string`）或已存在的有效导入。
    * *输出状态*: **⚪️ 无需处理**
    * *操作*: 忽略。
